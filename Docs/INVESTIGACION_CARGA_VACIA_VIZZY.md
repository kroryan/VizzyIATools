# Investigation of the Bug: Juno Loads the XML but the Vizzy Editor Appears Empty

## Result

The main problem was not the BOM or the XML declaration. The real cause was structural:

- the generator was writing `CustomInstruction` nodes inside `<Expressions>`
- Juno expects `CustomInstruction` nodes to live inside additional root-level `<Instructions>` blocks
- `<Expressions>` must be reserved for `CustomExpression`

This caused:

- XML that was parseable
- local validation that looked correct
- an internal `NullReferenceException` in the Vizzy editor when it tried to rebuild the nodes

## Evidence collected

### 1. The file had the correct BOM

It was verified that the generated XML started with:

- `EF BB BF`

This rules out the classic empty-file bug caused by a missing BOM.

### 2. The XML declaration was not the issue

It was compared against original files in `UserData/FlightPrograms`, and several of them also start with:

- `<?xml version="1.0" encoding="utf-8"?>`

So the XML header does not explain the failure.

### 3. `Player.log` showed an editor failure, not an XML parser failure

`Player.log` contained:

```text
NullReferenceException: Object reference not set to an instance of an object
at Assets.Scripts.Vizzy.UI.VizzyUIScript.LoadFlightProgram
```

That indicates:

- the XML opens
- but the Vizzy editor cannot rebuild its internal structure

### 4. Key comparison with a valid original XML

In the original `Universal Vizzy Mission 2 - Enhanced.xml`, `CustomInstruction` nodes are serialized like this:

```xml
<Instructions>
  <CustomInstruction ... />
  <SetVariable ... />
  <If ... />
</Instructions>
<Instructions>
  <CustomInstruction ... />
  <For ... />
</Instructions>
<Expressions>
  <CustomExpression ... />
</Expressions>
```

In the broken version generated by the tool, the pattern was:

```xml
<Instructions>
  ... main thread ...
</Instructions>
<Expressions>
  <CustomInstruction ... />
  <If ... />
  <SetVariable ... />
</Expressions>
```

That format does not match what Juno uses for editable programs.

## Second issue detected

The tool assumed:

```python
USERDATA = Path(__file__).parent
```

But in this repository `vizzy_tool.py` lives inside:

- `UserData/VizzyIATools`

not directly inside:

- `UserData`

Consequence:

- the tool could read and write in `VizzyIATools/FlightPrograms`
- but Juno actually uses `UserData/FlightPrograms`

This was fixed so it automatically detects whether the real `UserData` directory is the parent directory.

## Changes applied

### 1. Fix in `vizzy_tool.py`

`VizzyProgram` serialization was changed:

- `add_custom_instruction(...)` no longer places those blocks in `_expressions`
- now each custom instruction is stored as its own thread
- `to_element()` emits:
  - one `<Instructions>` block for the main thread
  - one extra `<Instructions>` block per `CustomInstruction`
  - one final `<Expressions>` block only for custom expressions

### 2. Path fix

`vizzy_tool.py` now detects:

- whether `FlightPrograms` and `CraftDesigns` are in the current directory
- or whether they are in the parent directory

That makes the tool point to the game's real `UserData`.

### 3. Generator fallback when the file is locked

If `Universal Orbit 200km - Auto.xml` is locked by Juno, the generator saves:

- `Universal Orbit 200km - Auto FIXED.xml`

This prevents the old broken file from blocking the corrected version from being written.

## State after the fix

The correct structure of the new program is:

- `Variables`
- main `Instructions`
- `Instructions` for `Auto Stage`
- `Instructions` for `Update Orbit State`
- empty `Expressions`

That pattern matches the original Juno XML files that use `CustomInstruction`.

## Conclusions

The two real failures were:

1. Incorrect `CustomInstruction` serialization
2. Incorrect `UserData` folder resolution

The first issue breaks the Vizzy editor even when the XML is parseable.
The second causes confusion because the tool can write a copy into a folder the game does not use.

## Practical next step

Run:

```bash
python generate_universal_200km_orbit.py
```

Then, in Juno, open:

- `Universal Orbit 200km - Auto FIXED.xml` if the old file was still locked
- or `Universal Orbit 200km - Auto.xml` if it could be overwritten
